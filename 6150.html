Background

Background

DEMs and orthomosaics usually come from LiDAR, but Structure-from-Motion (SfM) photogrammetry can make the same products from regular imagery, no LiDAR sensor or drone needed. The tradeoff is compute. Modern cameras produce huge numbers of high resolution frames, and processing them into a dense point cloud can take a laptop days or weeks.

This project comes out of Lower Cataract Canyon, where the Colorado River meets Lake Powell. Drones are banned in Glen Canyon National Recreation Area, so ground-based SfM is the only realistic way to map the eroding Dominy Formation. I built a headless Metashape pipeline (no GUI, all Python) that runs through SLURM on the University of Utah's Center for High Performance Computing (CHPC) and cuts a week-plus laptop job down to hours.

Study area

The Dominy Formation at Gypsum Canyon

The Dominy Formation is unconsolidated sediment that built up as the Colorado River slowed into Lake Powell. When the reservoir dropped in the early 2000s, tens of millions of cubic meters of submerged sediment were exposed, and the river is now cutting back through it. Field teams from the University of Utah, UGS, USGS, the Glen Canyon Institute, and the Returning Rapids Project visited Lower Cataract Canyon October 15–17, 2025. This project uses the October 15 dataset from the mouth of Gypsum Canyon, collected under Dr. Cari Johnson.

Data

Multi-distance video capture

Four videos film the same outcrop from increasing distances. The close passes capture fine texture, and the far passes tie the scene together and cover the wash bed where the GNSS control points sit. I extracted one frame per second (every 30th frame), which gave 860 PNG frames. The frames have no GPS metadata, so everything runs in local coordinates.

Methods

Validate, then scale

Everything had to run without a GUI, scripted against Metashape's Python API and submitted through SLURM. Before running the full dataset, I pushed a 55-photo test set through the whole pipeline. It took 16 minutes and confirmed the job scripts, file paths, and API calls all worked before I committed to the 53-hour run. Metashape's API changes between versions, and headless runs on the CHPC give you almost nothing to debug with, so I checked attribute names in an interactive Python shell before finalizing the scripts.

The pipeline's five steps, condensed:

# Step 1: Align cameras (content-based matching, no GPS)
chunk.matchPhotos(generic_preselection=True, reference_preselection=False)
chunk.alignCameras()

# Step 2: High-resolution depth maps
chunk.buildDepthMaps(downscale=1, filter_mode=Metashape.MildFiltering)

# Step 3: Dense point cloud (~20M points)
chunk.buildPointCloud(source_data=Metashape.DepthMapsData, point_colors=True)

# Step 4: Orthomosaic, using the point cloud as the surface
chunk.buildOrthomosaic(surface_data=Metashape.DataSource.PointCloudData,
                       blending_mode=Metashape.MosaicBlending, fill_holes=True)

# Step 5: Export as TIFF
chunk.exportRaster(path=ORTHO_PATH, source_data=Metashape.OrthomosaicData,
                   image_format=Metashape.ImageFormatTIFF)

Full SLURM and Python scripts are documented in the project report; the job ran on a single Kingspeak node (8 cores, 64 GB RAM).

Results

Results

All 860 frames aligned, most on the first attempt, which points to good overlap between the four videos. The depth maps produced a dense cloud of roughly 20 million points, densest near the outcrop and thinner in the distant Video 62 frames. The point cloud then served as the surface for the orthomosaic, which came out seamless and captures the sediment layering well.

The full run finished in 53 hours with no crashes or timeouts. The same job would take an estimated 10 to 15 days on a personal computer. Only two GNSS points landed inside the videos, and three are needed to constrain rotation, so the models are not fully georeferenced. They still work as outcrop references for tracing erosion features and studying the layering.
